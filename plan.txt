# TaskRoot — Requirements Specification

## Overview

TaskRoot is an opinionated, single-user task management system for Windows, with Android support deferred to a later phase. It is always running while the computer is on. It is minimalist in aesthetic and deliberately structured in workflow, guiding the user through a fixed daily cycle rather than offering a free-form interface.

---

## Platform

**Primary:** Windows 11.

**Deferred:** Android. When added, Android supports adding and reading tasks only — none of the workflow phases exist on Android. The data layer must be designed now to accommodate bidirectional sync without significant refactoring. Sync mechanism is unresolved; SQLite with a periodic JSON export is the likely bridge for file-based sync tools like Syncthing.

---

## Tech Stack

- Python, managed with `uv`
- PyQt for window management, system tray, and OS-level calls
- `pywebview` for rendering the UI
- Svelte for all visual UI (runs inside the webview)
- SQLite for storage

The PyQt/pywebview layer handles: window lifecycle, tray icon, startup registration, OS notifications, idle detection. It exposes a Python API that the Svelte frontend calls via the webview bridge. The Svelte layer handles everything the user sees.

---

## Always-On Behaviour

The application starts on system startup. Closing the window does not terminate the process — it minimises to the system tray. The tray icon is always present while the computer is on. **[OPEN]** The keyboard shortcut to summon/hide the main window needs to be defined.

---

## The Daily Cycle

A "day" begins at **04:00** and ends at 03:59 the following morning.

The cycle has five phases in order: **Capture → Clarify → Plan → Do → Shutdown.**

A day is considered **initialised** once the user has reached Do for the first time that day.

On launch:
- If uninitialised: begin at Capture and guide forward through the phases.
- If initialised: drop directly into Do.

The user may re-enter Capture and Clarify mid-day. This is accessible via Reorient.

---

## UI Principles

- Every screen is highly minimalist.
- The titlebar contains no app name, no minimise button, no close button.
- Navigation is keyboard-driven. Shortcuts are single-key or two-key combinations without Ctrl chording. The interface is **not modal** (no normal/insert mode distinction).
- **[OPEN]** The full keyboard shortcut map is to be defined during prototyping.
- The aesthetic must induce calm. Clean. No visual noise.

## Data Model

### Task

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | |
| `name` | string, max 20 words | Mandatory. Must begin with a verb. Hard rejected otherwise. |
| `description` | string, max 100 words | Optional. |
| `work_date` | date | Optional. When the user intends to do it. |
| `deadline` | date | Optional. When it must be done. |
| `tags` | list of tag IDs | Optional. Independent of parent task. |
| `parent_id` | task ID | Optional. Unlimited nesting depth. Projects do not exist; nested tasks serve this role. |
| `recur_rule` | RecurRule or null | Optional. See below. |
| `expected_duration` | minutes | Optional. |
| `is_low_thought` | bool | Optional. Set during Clarify. Determines which list the task appears in on the Do screen. |
| `created_at` | datetime | |
| `completed_at` | datetime | Null if incomplete. |

`actual_duration` is never stored on the task directly. It is derived by summing the duration of all TimeSession records linked to this task.

**Overdue behaviour:** If `work_date` passes and the task is not complete, it enters an overdue state. It does not auto-roll. It remains in the overdue pile until rescheduled or completed.

**Verb validation:** Task name is checked against a curated English verb list (sourced from a standard dictionary dataset) on save. If the first word is not a recognised verb, the task is hard rejected with an inline error.

### Recurrence

Recurrence uses an **instance model.**

A task with a `recur_rule` is a **template task.** It is never completed itself. Each day the rule fires, a new **instance task** is created, linked to the template via `parent_id`. The instance is a normal task in every respect and carries its own `completed_at`, `actual_duration`, and `work_date`.

Instance tasks are created **at the start of the day they are due** (i.e. at 04:00), not pre-generated in advance. However, the calendar views (month, week, day) render future projected occurrences by computing them from the recurrence rule at display time — no task objects are created for future dates.

**RecurRule fields:**
- `type`: `daily` | `weekly` | `monthly`
- `interval`: integer (e.g. every 2 days)
- `days_of_week`: list (for weekly rules, e.g. every Monday and Wednesday)
- `day_of_month`: integer (for monthly rules)

Recurring task instances are **hidden from the calendar by default.** A toggle shows them.

### Event

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | |
| `name` | string, max 20 words | Mandatory. |
| `description` | string, max 100 words | Optional. |
| `start` | datetime | Required. May be date-only (no time component). |
| `end` | datetime | Optional. May be date-only. |

Events are not tasks. They cannot be completed, assigned to a project, or time-tracked.

### TimeSession

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | |
| `task_id` | task ID | |
| `started_at` | datetime | |
| `ended_at` | datetime | Null if session is still active. |

One active TimeSession may exist at a time. Starting a new one ends the current one.

### Distraction

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | |
| `text` | string | The distraction as typed by the user. |
| `logged_at` | datetime | |

Distractions are their own first-class entity. New fields may be added in the future. The distraction inventory on the Do screen is a view into this table.

### Tag

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | |
| `name` | string | |

### Extensibility

The storage layer must handle unknown fields gracefully — storing and round-tripping them without dropping them. New fields may be added to any entity at any time.

## Phase: Capture

Entry points: start of an uninitialised day, or re-entry from Reorient mid-day.

The user adds tasks and events freely. The only mandatory field during Capture is the task name. All other fields are optional and may be filled in during Clarify.

While typing a task name, similar existing tasks are surfaced in real time using fuzzy string matching, to prevent duplicate entries.

Capture ends when the user explicitly advances. No minimum task count is required.

---

## Phase: Clarify

Tasks are presented one at a time. The user reviews and fills in optional fields. The minimum action to pass a task through is a single keypress to accept defaults. The intent is speed, not thoroughness.

The `is_low_thought` field is set here.

---

## Phase: Plan

**Layout:** Month calendar view, with the ability to navigate to week and day views.

The calendar shows:
- Event start and end dates.
- Task deadlines.
- Task work dates.
- Recurring instance projections (hidden by default, toggle to show).

**Today view:** Two lists side by side:
- Low-thought tasks (`is_low_thought = true`) — for breaks and downtime.
- Thought-intensive tasks (`is_low_thought = false`) — for focused work.

---

## Phase: Do

### Main Screen

All sections are collapsible.

**Task lists**
Two lists: low-thought and thought-intensive, matching the Plan view. Starting a task begins a TimeSession.

**Time tracker**
Manual start/stop per task. Only one task may be active at a time.

Check-in reminder: a notification fires 50 minutes after a task was last started. If not dismissed, subsequent notifications fire every minute. This handles the case where the user has walked away or dissociated.

Idle detection: **[OPEN]** system idle threshold and exact behaviour (pause timer silently, or prompt?) to be defined.

**Music player**
Embedded. Details to be defined.

**Calculator**
Placeholder only in this phase. Will embed Desmos or WolframAlpha in a later phase.

**Note-taker**
Plain text field. Cleared at the start of each new day.

**Persistent text area**
Plain text field for motivational content or reference material. Persists across days indefinitely.

**Distraction inventory**
Input field. User types a distraction and hits Enter; it is saved as a Distraction record with the current timestamp. The inventory displays all distractions logged today, in reverse chronological order.

**Obsidian links**
A global list of links to Obsidian notes, configured once in Settings. Shown on the Do screen for quick access. Not per-task.

**Break types**
Logged as TimeSession records against special system-level break tasks, not against user tasks.

| Name | Default duration |
|---|---|
| Piss | 5 min |
| Talk | 20 min (extendable) |
| Shit | 20 min |
| Nap | 20 min |
| Walk | 30 min |

### Floating Widget

A small, titlebar-less window visible only during the Do phase. Default position: right screen border, above the taskbar. Position is user-configurable in Settings.

Displays:
- If a task is active: the task name and a running stopwatch showing time spent on the current session.
- If no task is active: the text "No task active."

---

## Phase: Reorient

The most comprehensive screen in the application. Accessible at any point after initialisation. Entry is via a button; it does not interrupt the Do phase but sits alongside it.

Contains:
- Add new tasks and events (re-enters Capture flow inline).
- Month calendar view.
- Week view.
- Day view.
- Statistics. **[OPEN]** Specific statistics to be defined during prototyping.
- Settings.

This screen is the only place Settings are accessible.

**[OPEN]** Full layout to be defined via prototype.

---

## Phase: Shutdown

Shutdown is the final phase of the day. It is initiated explicitly by the user via a button, in the same way other phase transitions are triggered. It is not time-triggered.

If the computer is shut down or the application is killed before the Shutdown ritual is completed, **only the ritual data is lost** (reflections, rating). All task completions, time sessions, and distractions logged during the day are already persisted to SQLite and are not affected.

**Shutdown screen contents, in order:**

1. **Day visualisation** — single-column timeline view (Google Calendar style). Populated automatically from TimeSession records. Shows tasks, breaks, and events as labelled blocks.
2. **Task table** — all tasks touched today, including those with no tracked time. Columns: task name, estimated time (`expected_duration`), actual time (sum of TimeSessions).
3. **Impact rating** — "How would you rate the impact of your work today?" 1–5 stars.
4. **Focus summary** — total minutes with an active TimeSession today; total minutes across all work (including untracked).
5. **Reflection notes** — four text fields:
   - What did you work on today?
   - What worked well?
   - What didn't work well?
   - What should change tomorrow?

---

## Non-Functional Requirements

**Performance:** Task operations, navigation, and search must feel instant. No perceptible lag.

**Resilience:** All task, session, and distraction data is written to SQLite immediately on change (write-on-change, not on a timer). The Shutdown ritual reflections are the only data that can be lost to an unexpected shutdown.

**Modularity:** Each phase is implemented as an isolated module. A bug or exception in one phase must not corrupt data or crash another. High cohesion, low coupling throughout. Pydantic models for all entities.

**Aesthetic:** Calm, clean, minimal. Every element on screen must earn its place.
